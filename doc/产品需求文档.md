# 产品需求文档：EatLog

## 0. 产品定义

| 项目 | 内容 |
|------|------|
| **产品名称** | EatLog |
| **产品类型** | iOS/Android 移动应用 |
| **核心目标** | 通过结构化饮食记录与身体反应追踪，利用统计分析帮助用户发现食物与身体不适的关联性，辅助管理肠道健康。 |

## 1. 业务核心逻辑

所有功能围绕以下数据闭环构建：

```
记录（因） → 追踪（果） → 分析（关联）
```

## 2. 功能性需求详解

### 模块一：今日记录首页（High Priority）

**目标**：使用户能一眼概览今日饮食与状态，快速进入记录。

| 需求项 | 详细说明 |
|--------|----------|
| **时间轴卡片** | 按时间倒序展示每餐记录。卡片包含：用餐时间、食物名称/照片、核心属性标签（辣度、油度、冷热）。 |
| **今日汇总** | 顶部展示"主食占比（饼图/环形图）"和"今日推荐/预警"（例如：今日脂肪摄入偏高）。 |
| **快捷入口** | 底部或悬浮按钮：快速添加"正餐"、"饮品"、"零食"、"反应症状"。 |
| **健康提示** | 展示"当前异常反应"或"已匹配的预警列表"，例如："建议避免油炸与冷饮混搭"。 |

### 模块二：新增记录（High Priority）

**目标**：将主观饮食信息转化为结构化数据。

| 需求项 | 详细说明 |
|--------|----------|
| **基础信息** | 食物名称（文本输入）、拍摄/上传照片（可选，作为辅助）。 |
| **温度维度** | 单选：冰镇 / 常温 / 温热 / 烫。 |
| **口感维度** | 单选：清淡 / 适中 / 微辣 / 辛辣 / 咸 / 甜。 |
| **油腻度** | 单选：无油 / 清淡 / 适中 / 油腻 / 重油。 |
| **烹饪方式** | 多选（可多选组合）：蒸 / 煮 / 炒 / 烤 / 炸 / 煎 / 生食 / 炖。 |
| **主食类型** | 单选：米饭 / 面条 / 馒头 / 面包 / 肉类 / 海鲜 / 其他。 |
| **饮品** | 可选录入，类型：奶茶 / 牛奶 / 咖啡 / 茶 / 冰水 / 白水。 |
| **肠胃反应记录** | 可与用餐记录独立，填写内容：开始时间、结束时间、症状类型（腹泻/胀气/绞痛/反酸）、严重程度（1-5 级）。 |
| **备注栏** | 自由文本，用户补充"今天很累"或"吃了药"等语境信息。 |
| **快捷键** | "复刻上一餐"功能，一键复制上一餐的标签与属性，节省录入时间。 |

### 模块三：统计分析（Medium Priority）

**目标**：提供周期性的数据可视化与趋势分析。

| 需求项 | 详细说明 |
|--------|----------|
| **周报/月报** | 以柱状图展示：用户一周内"食用频率"排名（如火锅出现 5 次，炒菜 10 次）。 |
| **敏感食物预警列表** | 核心功能。统计周期内，与"身体反应"高相关性的食物/组合，按"不适指数"从高到低列表展示。例如：冰奶茶（10次不适）/ 油炸食品（8次不适）。 |
| **关联度算法** | 核心逻辑：识别时间窗内（饭后 0.5 - 6 小时）的饮食与反应记录，计算置信度。 |
| **历史详情** | 点击图表中的某一天，可下钻查看该日详情。 |

### 模块四：个人中心/设置（Low Priority）

**目标**：管理用户数据与偏好。

| 需求项 | 详细说明 |
|--------|----------|
| **标签管理** | 允许用户自定义"烹饪方式"或"口感"分类（如增加"芥末味"）。 |
| **数据导出** | 导出 Excel 或 CSV，供用户自己交给医生进行医疗咨询。 |
| **提醒设置** | 开启饭后 2-4 小时"是否出现不适？"的定时推送，防止用户忘记记录反应。 |

## 3. 非功能性需求（NFR）

### UI/UX 要求

- 设计语言需**清爽、医学极简、无广告**，配色以低饱和度的绿色/白色为主。
- 减少点击层级，主要操作应在 **3 次点击内**完成。

### 数据持久化

- 确保**云端备份**（使用 Firebase 或类似服务），防止用户换手机丢失多个月的数据。
- 提供**离线录入模式**（网络恢复后自动同步）。

### 隐私合规

- 数据涉及健康隐私，必须严格遵循 GDPR（如适用）或当地法律，数据需匿名化存储，用户拥有删除权。

### 算法响应

- 预警列表的生成需在每次记录新数据后**实时刷新**，而不是仅在月末生成。
