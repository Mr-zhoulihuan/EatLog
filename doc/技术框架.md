```
┌─────────────────────────────────────────┐
```

## 2. 技术选型

### 2.1 核心框架

| 选项 | 说明 |
|------|------|
| **React Native 0.76+** | 跨平台移动框架，iOS/Android 一套代码 |
| **Expo (Managed Workflow)** | RN 的上层封装，简化构建、OTA 更新、推送等基础设施 |
| **TypeScript** | 类型安全，降低大型项目维护成本 |

> **选型理由**：React Native + Expo 生态成熟，社区活跃，能快速交付 MVP。TypeScript 保证代码质量。

### 2.2 UI & 样式

| 选项 | 用途 |
|------|------|
| **NativeWind (TailwindCSS for RN)** | 原子化样式，开发效率高 |

| **React Navigation** | 页面路由与导航 |

### 2.3 状态管理

| 选项 | 用途 |
|------|------|
| **Zustand** | 轻量全局状态（用户偏好、当前餐记录草稿） |

### 2.4 本地存储 & 离线

## 3. 数据模型设计

### 3.1 核心表结构（SQLite 本地）

```sql
-- 用餐记录
CREATE TABLE meal (
  id            TEXT PRIMARY KEY,
  food_name     TEXT NOT NULL,
  photo_uri     TEXT,
  temperature   TEXT,            -- 冰镇/常温/温热/烫
  taste         TEXT,            -- 清淡/适中/微辣/辛辣/咸/甜
  oiliness      TEXT,            -- 无油/清淡/适中/油腻/重油
  cook_methods  TEXT,            -- JSON: ["炒","炸"]
  staple        TEXT,            -- 米饭/面条/...
  drink         TEXT,            -- 饮品类型
  drink_custom  TEXT,
  note          TEXT,
  created_at    INTEGER NOT NULL, -- Unix timestamp
  updated_at    INTEGER NOT NULL,
  synced        INTEGER DEFAULT 0
);

-- 烹饪方式关联（多对多）
CREATE TABLE meal_cook_method (
  meal_id   TEXT NOT NULL,
  method    TEXT NOT NULL,
  PRIMARY KEY (meal_id, method)
);

-- 身体反应记录
CREATE TABLE symptom (
  id          TEXT PRIMARY KEY,
  meal_id     TEXT,                -- 关联的用餐记录（可为空，独立记录）
  start_time  INTEGER NOT NULL,
  end_time    INTEGER,
  type        TEXT NOT NULL,       -- 腹泻/胀气/绞痛/反酸
  severity    INTEGER NOT NULL,    -- 1-5
  note        TEXT,
  created_at  INTEGER NOT NULL,
  synced      INTEGER DEFAULT 0
);

-- 自定义标签
CREATE TABLE custom_label (
  id     TEXT PRIMARY KEY,
  category TEXT NOT NULL,  -- cook_method / taste / symptom_type
  value  TEXT NOT NULL,
  color  TEXT
);
```

## 5. 关联度算法（核心逻辑）

```
时间窗口：饭后 0.5h - 6h 内出现不适 → 视为潜在关联
不适指数 = log(关联次数 + 1) × 平均严重程度 × 近期因子

近期因子 = 1 + (最近30天内出现次数 / 总次数) × 0.5
```

触发时机：
- 每次新增 symptom 记录后，实时重新计算所有食物/组合的不适指数
- 结果缓存至本地，前端直接读取排序展示

## 6. 离线同步策略

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│  本地录入  │ ──→    │  SQLite  │ ──→    │  Firebase │
│  (离线)   │         │ (标记脏数据)│        │  (在线同步)│
└──────────┘         └──────────┘         └──────────┘
                           ↑                    │
                           └──── 双向合并 ──────┘
```

- 写入操作：优先写本地 SQLite，`synced=0` 标记待同步
- 网络恢复后：批量上传未同步记录，拉取远端新增数据
- 冲突处理：以 `updated_at` 较新者为准（Last Write Wins）
